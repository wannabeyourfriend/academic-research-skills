## 2. Five v3.10 delta items

### Delta 1 — arXiv API resolver

The current arXiv surface is a preprint-risk heuristic at `scripts/contamination_signals.py` plus venue-string matching in `bibliography_agent`. There is no API call to verify a cited arXiv ID resolves to a real preprint with matching metadata. v3.10 adds:

- **`scripts/arxiv_client.py`** — new module mirroring the `crossref_client.py` shape. Implements DOI-lookup-equivalent for arXiv IDs (`arxiv_id_lookup`), title-fallback search, exponential backoff, and the same unmatched semantics (`resolve_arxiv_unmatched`) used by the v3.9.0 triangulation pipeline.
- **`scripts/contamination_signals.py` signal computation**: add `arxiv_unmatched` boolean computation to the per-citation signal pass (mirrors how `crossref_unmatched` + `openalex_unmatched` are computed today). This file scope is **signal computation only**; the matrix dispatch and formatter pass-through changes live in the surfaces below.
- **`shared/contracts/passport/literature_corpus_entry.schema.json` schema extension**: add optional `arxiv_unmatched` boolean field to `contamination_signals` alongside the v3.9.0 `crossref_unmatched` + `openalex_unmatched` triplet. The manual-entry not-rule extends from v3.9.0 `anyOf: [s2, openalex, crossref]` to include `arxiv`.
- **`academic-pipeline/agents/pipeline_orchestrator_agent.md` §3.6 finalizer matrix advisory tier extension**: v3.9.0's `k=0..3` advisory matrix extends to `k=0..4` to cover four-index triangulation outcomes. The annotation token vocabulary extends with `CONTAMINATED-ARXIV-UNMATCHED` (single-index k=1 on arxiv only) and `CONTAMINATED-QUADRANGULATION-UNMATCHED` (full four-index k=4), and the v3.7.3 / v3.9.0 PREPRINT composition rule extends to compose these new tokens too: `CONTAMINATED-PREPRINT+ARXIV-UNMATCHED` and `CONTAMINATED-PREPRINT+QUADRANGULATION-UNMATCHED` (PREPRINT prefix first per canonical token order). The terminal gate behavior remains unchanged from v3.7.3 / v3.9.0 (advisory only; no new gate-refuse classes from triangulation).
- **`scripts/claim_audit_finalizer.py` dispatch**: implement the matrix extension defined by the orchestrator above. The 8-row finalizer matrix is **not** extended (no new rows); the suffix tokens compose at the annotation-rendering layer per the v3.9.0 advisory-suffix pattern.
- **`academic-paper/agents/formatter_agent.md` pass-through allowlist**: extends from 9 v3.9.0 tokens to 13 v3.10 tokens (adds the 4 new tokens above: `CONTAMINATED-ARXIV-UNMATCHED`, `CONTAMINATED-QUADRANGULATION-UNMATCHED`, `CONTAMINATED-PREPRINT+ARXIV-UNMATCHED`, `CONTAMINATED-PREPRINT+QUADRANGULATION-UNMATCHED`). Refusal rules 1-10 unchanged.

Out of scope for Delta 1: arXiv full-text resolution. The arXiv API verifies metadata and existence; it does not feed the v3.10 #183 epistemic-status extractor's Tier 2 pdftotext pathway. A user who wants Tier 2 verification on an arXiv preprint must still provide a local PDF via `source_pointer`. This separation keeps the citation gate (this spec) orthogonal to the epistemic-status gate (#183).

### Delta 2 — Persistent verification cache

Crossref / OpenAlex / S2 clients today implement per-request backoff but no persistent caching across pipeline runs. The same paper cited in multiple drafts re-hits the API every time. v3.10 adds:

- **`scripts/verification_cache.py`** — new module managing a local SQLite-backed cache (file at `~/.cache/ars/verification.db`, configurable via `ARS_VERIFICATION_CACHE_PATH`). Cache key is `(citation_key, resolver_name, query_form)` where `resolver_name ∈ {crossref, openalex, semantic_scholar, arxiv}` and `query_form` is canonical-form of the DOI / arXiv ID / title-query string. Cache value is the resolver's structured response plus a verification timestamp.
- **Cache invalidation policy**: TTL 90 days from verification timestamp; manual invalidation via `ars-cache-invalidate <citation_key>` slash command (added under deliverable 6 below). The 90-day window is a guess; OQ-1 lists it as deferred for empirical tuning.
- **Cache hit semantics**: a cache hit returns the structured response immediately and skips the API call. Cache miss or expired entry triggers the live API call; on success, the cache is populated. The four resolvers (crossref, openalex, semantic_scholar, arxiv) share the cache schema but key their entries by `resolver_name`.
- **Concurrency**: SQLite WAL mode for single-writer-many-readers safety. The audit pipeline is single-process; concurrent runs (multi-user shared cache) are out of scope for v3.10.

Cache integration touches the four existing resolver modules (crossref_client.py, openalex_client.py, the S2 lookup in `bibliography_agent`, the new arxiv_client.py) with a uniform wrapper: each resolver's public entry point grows a `cache: VerificationCache | None = None` parameter and, when present, consults the cache before making the network call. v3.8 calibration runner and audit pipeline tests pass a `cache=None` to preserve byte-equivalent behavior on test inputs.

### Delta 3 — Promote `ARS_CLAIM_AUDIT=1` from opt-in to blocking default

v3.8 ships `ARS_CLAIM_AUDIT=1` as default OFF per v3.8 spec §5 mode flag rationale ("ramp-on plan deferred to post-calibration evidence"). v3.9.x runs accumulated the empirical evidence the v3.8 rationale required (calibration runner at `scripts/claim_audit_calibration.py`, T-C1/T-C2/T-C3 thresholds passing on the 20-tuple gold set per `academic-pipeline/references/claim_audit_calibration_protocol.md`). v3.10 promotes the gate.

The promotion is **policy-only** (no code-path change beyond the flag default flip and one branch deletion):

- **`academic-pipeline/SKILL.md` flag rationale section**: the v3.8 paragraph at line 25 ("v3.8 (opt-in): Set `ARS_CLAIM_AUDIT=1` ...") is replaced with a v3.10 paragraph stating the gate is on by default and the flag now exists only for opt-out backwards compatibility (`ARS_CLAIM_AUDIT=0` skips the gate; default behavior dispatches the audit regardless of the env-var presence).
- **`academic-pipeline/agents/pipeline_orchestrator_agent.md` §3.6 dispatch**: the conditional dispatch (`if env('ARS_CLAIM_AUDIT') == '1'`) becomes unconditional dispatch; the explicit opt-out branch (`if env('ARS_CLAIM_AUDIT') == '0': skip`) is added.
- **`scripts/test_claim_audit_pipeline.py`** updates: the existing flag-on test cases become the default, the flag-off test cases become the opt-out path. T-C1 / T-C2 / T-C3 calibration thresholds and the 20-tuple gold set are unchanged.

The promotion's load-bearing safety property: the formatter HIGH-WARN refusal classes (rules 6-10 per v3.8 spec) now block production output by default for the five classes the v3.8 audit already detected (HIGH-WARN-CLAIM-NOT-SUPPORTED, HIGH-WARN-NEGATIVE-CONSTRAINT-VIOLATION, HIGH-WARN-FABRICATED-REFERENCE, HIGH-WARN-CLAIM-AUDIT-ANCHORLESS, HIGH-WARN-CONSTRAINT-VIOLATION-UNCITED). Per #198 acceptance criterion, this delta is what makes the v3.10 verification layer the **default critical path** rather than an opt-in advisory pass.

**Formatter REFUSE list extension (v3.10 rule 12):** Delta 3 adds a new formatter refusal class to `academic-paper/agents/formatter_agent.md` `Cite-Time Provenance Hard Gate`, sibling to rules 6-10 (v3.8) and rule 11 (v3.10 silent-upgrade per #183 §6.1):

- **Rule 12 (NEW v3.10):** Refuse on any `citation_verification_summary[]` row where `lookup_verified == false`. Refusal annotation: `[UNVERIFIED CITATION — lookup_verified=false: citation_key {citation_key}, ref_slug {ref_slug}, resolvers_unmatched=[{list of resolver names that returned status=unmatched}]]`. Rule 12 has no dependency on reviewer agent verdict; it is a deterministic gate-refuse triggered by the unified summary produced by Delta 4.

This refusal class is what satisfies acceptance criterion B8 below. The rule is cross-referenced into the #183 formatter refusal list extension table at §6.1 so the two specs share a single source of truth for the v3.10 refusal surface. The `unresolvable` outcome does NOT trigger Rule 12 per OQ-4's advisory-pass default; only `false` does.

Backwards-compatibility note: v3.8 / v3.9.x corpora that contain unreviewed audit findings stored in the passport pre-v3.10 are not retroactively re-audited. v3.10 applies only to new pipeline runs.

### Delta 4 — Unified `lookup_verified` + `anchor_present` per-citation status surface

Today the citation-level verification signals are split across three schemas: anchor presence lives in v3.7.3 anchor markers (read at `formatter_agent`'s `Cite-Time Provenance Hard Gate`), Crossref/OpenAlex/S2 unmatched signals live in `contamination_signals`, and claim-audit verdicts live in `claim_audit_result`. A consumer that wants to know "is this citation verified or not" must read three schemas and apply join logic. v3.10 adds a derived view:

- **`shared/contracts/passport/citation_verification_summary.schema.json`** — new aggregate schema. One entry per (citation_key, ref_slug) pair. Required fields:
  - `citation_key`: joins to `literature_corpus_entry.citation_key`.
  - `ref_slug`: joins to the `<!--ref:slug-->` markers used in writer prose.
  - `lookup_verified`: enum of `{true, false, unresolvable}`. `true` iff at least one applicable resolver returned `status=matched`. `false` iff at least one applicable resolver returned `status=unmatched` AND no applicable resolver returned `status=matched` (i.e., positive evidence of non-existence from at least one source). `unresolvable` iff **every** applicable resolver returned `status=unreachable` AND none returned (`status=matched` OR `status=unmatched`) (i.e., the system could not adjudicate because every applicable resolver was offline). The classification is anti-fabrication-biased: a fabricated citation with bogus DOI that produces three `unmatched` + one `unreachable` is classified `false`, not `unresolvable`, because a single `unmatched` is positive evidence of non-existence and one transient outage does not cancel that evidence. `skipped` outcomes are excluded from the classification condition entirely; they reflect resolver applicability or policy (arXiv resolver `skipped` on non-arXiv citation per Delta 1; manual entry `skipped` per v3.7.3 `obtained_via: manual` exemption), not adjudication. When all applicable resolvers are `skipped` (e.g., a manual entry that no resolver attempts), the set of adjudicating resolvers is empty and `lookup_verified` defaults to `unresolvable`; this is the manual-entry-exempt case and matches the v3.9.x triangulation advisory tier. Downstream consumers needing to distinguish manual exemption from total outage read `resolver_outcomes.*.status` directly.
  - `anchor_present`: boolean. `true` iff the citation's v3.7.3 anchor marker has `kind ∈ {quote, page, section, paragraph}` (not `none`).
  - `verification_timestamp`: ISO-8601 UTC. When at least one resolver ran, this is the latest resolver timestamp. When all resolvers were skipped (manual entry) or all unreachable (outage), this is the attempt timestamp (when the summary was generated). Never null.
  - `resolver_outcomes`: object with one optional field per resolver (`crossref`, `openalex`, `semantic_scholar`, `arxiv`); each value is `{status: "matched" | "unmatched" | "unreachable" | "skipped", response_summary: string | null}`. The `status` enum is one-of (mutually exclusive states); replaces the three-boolean encoding from an earlier draft of this spec which permitted contradictory combinations. `matched` = resolver returned a match. `unmatched` = resolver ran and returned no match. `unreachable` = API outage / timeout. `skipped` = resolver did not run (manual entry per v3.7.3 exemption, or arXiv resolver on a non-arXiv citation per Delta 1).
- **Derivation point**: `scripts/citation_verification_summary.py` (new module) generates this aggregate at the Stage 4→5 boundary by joining the existing three sources. It runs after the v3.8 audit pipeline and before the formatter REFUSE pass.

This summary is the read interface for the §3.6 `verification_gate` contract (per #183 spec). The other two children of #198 reference this surface:

- **#183 epistemic-status gate** reads `anchor_present` to know whether the row is Shape C.1 (anchorless skip per #183 §5.7.2) or eligible for the v3.10 emitted-status inference.
- **#184 eval harness** reads `lookup_verified` as the per-citation truth signal for the citation-extraction subset of its gold set.

`lookup_verified` semantics intentionally omit a "partial match" or "conflict" value. v3.9.0's triangulation already encodes those nuances in `contamination_signals.k_unmatched`; the v3.10 summary collapses them to a binary "did at least one external index agree." A consumer that needs the fine-grained breakdown reads `resolver_outcomes` directly.

### Delta 5 — `verification_gate` standalone API surface

#198's acceptance criterion 4 requires "each child gate emits output in the shared shape so downstream consumers can read all three uniformly." Delivering this on the citation side requires extracting the gate logic from its current Stage 4→5 finalizer-step encoding into a callable API.

- **`scripts/verification_gate/__init__.py`** — new package wrapping the citation verification logic. Public functions:
  - `verify_citation(citation_key, manifest, corpus, cache=None) -> CitationVerificationOutcome` — runs the full triangulation (Crossref + OpenAlex + S2 + optional arXiv) and returns a structured outcome (`lookup_verified`, `anchor_present`, `resolver_outcomes`, `verification_timestamp`).
  - `verify_passport(passport, cache=None) -> list[CitationVerificationOutcome]` — batch helper running `verify_citation` over every (citation_key, ref_slug) pair in the passport.
- The new API does **not** duplicate the v3.8 audit pipeline; it composes the existing crossref_client / openalex_client / S2 lookup / new arxiv_client (Delta 1) and writes the unified summary schema (Delta 4). The v3.8 audit pipeline's audit-finding writers (which produce `claim_audit_result` rows) continue to call into the same underlying resolvers; the new API and the v3.8 audit are two callers of the same lower-layer infrastructure rather than duplicate implementations.
- The API is invoked at three points in v3.10:
  1. Stage 4→5 audit pipeline (existing, now unconditional per Delta 3).
  2. Standalone command-line entry point for ad-hoc verification (`scripts/verify_passport.py`, callable as `python -m scripts.verify_passport <passport.yaml>`).
  3. Future consumers — the #184 eval harness reads via this API during gold-set comparison.

The API stability contract: function signatures and return shapes are frozen at v3.10.0 and only additively extended in patch releases. Removing a field from `CitationVerificationOutcome` requires a minor-version bump and a deprecation cycle.

## 4. Out of scope

- **Replacing the v3.8 audit pipeline.** v3.10 #182 adds a parallel verification API surface; the v3.8 `claim_audit_pipeline.py` continues to run with its existing 8-row finalizer dispatch. v3.8 rows and the new `citation_verification_summary[]` aggregate are independent passport surfaces; downstream consumers read whichever fits their use case.
- **Anchor-aware citation insertion during draft generation.** Already shipped in v3.7.3 three-layer emission; not a v3.10 deliverable.
- **Multi-user shared cache.** Concurrent runs sharing a single cache file are out of scope. The cache assumes single-process access; multi-process is a future follow-up if usage demands.
- **Conflict resolution between resolvers.** When Crossref says match and OpenAlex says unmatched, v3.10 reports both via `resolver_outcomes` and `lookup_verified` is `true` (at-least-one match). A "Crossref disagrees with OpenAlex on the matched paper's metadata" reconciliation is out of scope; v3.9.4 `citation_provenance` already covers the year-precision disagreement case for first-party verification, which is the only conflict type the existing surface handles.
- **`evidence_type` 6-tier biomedical refinement.** Out of scope per #198 umbrella (the v3.10 vocabulary work belongs to #183 and is locked to the 5-tier cross-disciplinary enum).

## 5. INVARIANTS

I-V1 through I-V15 from `2026-05-21-v3.10-183-epistemic-status-spec.md` §9 are not duplicated here. The following are #182-specific:

**C-V1 — Triangulation cardinality.** `contamination_signals.k_unmatched` reports the count of resolvers that returned unmatched among those that ran. With arXiv added, `k` ranges from 0 (all matched) to 4 (all four unmatched on a citation that has an arXiv ID); for citations without an arXiv ID, the upper bound is 3. The `arxiv_unmatched` field is optional in the schema and absent when the citation has no arXiv ID; lint asserts this absence-vs-`false` distinction.

**C-V2 — Cache transparency.** Cache hits and misses produce byte-equivalent outputs at the v3.10 API level. A consumer cannot distinguish a cache hit from a fresh API call by inspecting the returned `CitationVerificationOutcome` (other than via `verification_timestamp`, which carries the original verification time on cache hits).

**C-V3 — Flag back-compat (scoped).** The opt-out path (`ARS_CLAIM_AUDIT=0`) MUST suppress all v3.10-added passport aggregates introduced by this spec (`citation_verification_summary[]`, `verification_gate` standalone API output, the new `arxiv_unmatched` contamination signal field). When suppressed, the passport surface on existing v3.8 / v3.9.x aggregates (`claim_audit_results[]`, `contamination_signals` minus `arxiv_unmatched`, etc.) MUST be byte-equivalent to a v3.9.x run on the same input. The byte-equivalence guarantee is scoped to the existing aggregates, not to the passport as a whole; the v3.10-added aggregates are present-or-absent depending on the flag, not "byte-equivalent to v3.9.x absent state" (a v3.9.x run had no concept of these aggregates, so byte-equivalence is undefined).

**C-V4 — verification_gate API stability.** The `verify_citation` and `verify_passport` signatures are frozen at v3.10.0. Patch releases add optional keyword arguments only; minor releases may add new required fields with a deprecation cycle and migration script.

**C-V5 — Anchor-present semantics.** `anchor_present == true` is the strict complement of v3.7.3's anchor-none case. `anchor_kind ∈ {quote, page, section, paragraph}` maps to `true`; `anchor_kind == "none"` maps to `false`. No third value is permitted; if an unrecognized `anchor_kind` appears, the row is a v3.7.3 contract violation and the v3.8 audit pipeline rejects it before this summary runs.

## 7. Open questions deferred to implementation

**OQ-1 — Cache TTL value.** The 90-day cache TTL is a guess. Implementation may tune based on observed Crossref / OpenAlex / S2 / arXiv metadata churn rates. Tuning is a runtime parameter (env-var configurable), not a schema change.

**OQ-2 — arXiv ID format strictness.** arXiv supports both old-style IDs (`hep-th/9711200`) and new-style IDs (`2605.07723`). Client must accept both; whether to canonicalize old-style to new-style for cache key purposes is deferred.

**OQ-3 — Multi-user cache concurrency.** Single-process SQLite WAL is adequate for v3.10 production use. If multi-user shared cache demand appears, a follow-up issue scopes the change (likely an HTTP-backed cache service, not a shared file).

**OQ-4 — `lookup_verified == unresolvable` ramp-up policy.** When every applicable resolver is `unreachable` (total outage), or when every applicable resolver is `skipped` (manual entry exempt from all four), `lookup_verified` is `unresolvable` per Delta 4. Whether this state should block ship (gate-refuse) or pass (advisory) is deferred to operational observation. v3.10 default is advisory (pass), matching v3.9.x triangulation's advisory tier. Note that the partial-outage case (some `unmatched` + some `unreachable`) is classified `false` per the Delta 4 anti-fabrication bias, not `unresolvable`, so the advisory-pass default never extends to citations where at least one resolver returned positive evidence of non-existence.
